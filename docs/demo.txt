Product
----------

Who uses this? SRE/On call engineer
what will they use it for?  Triage issues
How does it help them? Reduce time to traiege/resolve issues

what is the problem they face currently?

1. Knowledge is fragmented across tools
 Runbooks live in Confluence etc., incident timelines live in Rootly/equivalent, fix notes live in Slack threads or Jira tickets.
  An on-call engineer facing an alert has no single place to ask "has this happened before, and how did we fix it?"

 Need to look at different information, spanning across different information systems
    - Runbooks
    - Postmorterms
    - Code docs
    - Logs
    - Metrics Dashboards

2. Format inconsistency. Runbooks aren't written to a standard format
    — some are markdown checklists, some are PDFs from a vendor, some are Word docs written by whoever was on-call that week. Any tool that hard-codes parsing for one format breaks the moment a new team's runbook arrives in a different shape — which kills onboarding speed for new teams/orgs.

3. Triage speed is the direct cost
    For an enterprise, incident MTTR (mean time to resolution) isn't just an engineering metric — it's tied to SLAs, customer trust, and (for regulated industries) compliance exposure. Every minute spent searching instead of fixing is a minute of measurable business cost.
